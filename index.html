Overview

I do have some other ideas, but this is what I am leaning towards right now. This web game will use an api to obtain words and hints. Currently I am thinking about making a hangman game. The user can roll dice to choose how many letters the given word will be. Then when a word is chosen, a category or hint of some kind will show up on screen. The api can provide words, hints, definitions, and examples. Guess the wrong letter or make an incorrect guess too many times and you loose. There is also a scoring and leaderboard system.

This application would use the Merriam Webster api to obtain words hints and definitions. It will be written in html, javascript, json, as well as css, and hosted on Azure. There will also be a database used to set and display scores.

Feature List

  • Use of API to obtain random words and hints
  • A way to display previously guessed letters that were wrong
  • Scoring system database
  • The ability for another person to choose what word they want you to have

Future Feature List

  • a timer, where you either need to get the word in x amount of time, or a timer where you need to guess a letter every x seconds

Technical Features

  • obtaining information from an api
  • retrieving data from a database
  • writing to a database

Advanced Technical Features

  • Asynchronous Loading of High Scores
  • Asynchronous Loading of an amount of words equal to the number rolled on the dice (so if a game goes to 10, I need to load in 10 words, and then I can store them in a list, and once one of the word is guessed, I can move on to the next one)
